# Bring-Your-Own AI Integration / Integration IA Externe

This guide shows how to connect OpenAI, Anthropic, Groq, Ollama, Hugging Face,
RunPod, or a custom model service to Yaatal Engine without changing Engine or
adding an AI namespace to `@yaatal/client`.

Ce guide montre comment connecter OpenAI, Anthropic, Groq, Ollama, Hugging Face,
RunPod ou un service modele sur mesure a Yaatal Engine sans modifier Engine et
sans ajouter de namespace IA a `@yaatal/client`.

## English

### V1 Contract

`@yaatal/client` V1 is the Engine control-plane client. It covers auth,
products, orders, delivery, BOBO commerce, search, notifications, and analytics.
It intentionally does not expose `client.ai` in V1.

AI is bring-your-own. The AI service owns inference, prompts, retrieval prompts,
tool selection, and response shaping. Engine owns identity, authorization,
commerce state, delivery state, notifications, analytics records, and every
state-changing action.

### Architecture

Use one of two paths, often both in the same app:

```text
App -> AI service -> @yaatal/client -> Engine
App -> @yaatal/client -> Engine
```

The AI-assisted path is for search help, summarization, recommendations, draft
cart construction, and order proposals. The direct app path is for normal app
screens, fallback behavior, and confirmed actions.

Keep model provider details behind your AI service:

```text
App
  sends user message and Engine JWT
AI service
  calls OpenAI, Anthropic, Groq, Ollama, Hugging Face, RunPod, or custom model
  calls @yaatal/client with the Engine JWT
Engine
  validates JWT, enforces ownership, writes business state only on confirmed calls
```

Do not send the Engine JWT to the model provider. The AI service can use the JWT
to call Engine through `@yaatal/client`, but prompts sent to providers should
contain only the minimum product/order context needed for the response.

### Boundary Rules

- AI owns inference and prompts.
- Engine owns state-changing actions.
- JWT identity comes from Engine auth. Do not trust `buyer_id`, `seller_id`, or
  user role claims generated by the model.
- User confirmation gates money, order creation, delivery confirmation, order
  cancellation, profile-sensitive mutations, KYC, address changes, and any
  action that changes commerce or delivery state.
- AI may search products, summarize results, explain tradeoffs, and draft an
  order proposal.
- AI must not directly create an unchecked order, confirm delivery, cancel an
  order, submit KYC, or mutate profile-sensitive data.
- Analytics may record AI-assisted events, but analytics is not business state
  and must not be used as proof that an action occurred.

### Operations Checklist

- `ENGINE_API_URL`: configure the AI service and app to use the same local,
  staging, or production Engine base URL.
- Auth token handling: pass the user's Engine JWT from the app to the AI service
  in `Authorization: Bearer <jwt>` when the AI service needs scoped Engine data.
  Never log JWTs, store them in model prompts, or send them to AI providers.
- AI provider keys: keep provider keys only in the AI service environment, for
  example `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GROQ_API_KEY`,
  `HUGGINGFACE_TOKEN`, `RUNPOD_API_KEY`, or a private local model URL.
- CORS and allowed origins: allow the app origin to call the AI service. Allow
  the app and AI service origins to call Engine only when they are expected
  browser clients. Server-to-server calls do not need browser CORS.
- Rate limits: set limits for app -> AI service, AI service -> provider, and AI
  service -> Engine. Treat provider retries and Engine mutations separately.
- Logging: redact JWTs, provider keys, phone numbers, addresses, profile fields,
  payment references, and raw prompts containing personal data.
- Fallback: when AI is unavailable, keep the direct app path working with
  `client.search`, `client.products`, `client.orders`, and `client.delivery`.
- No unchecked AI mutations: expose proposal endpoints and confirmation
  endpoints separately. Confirmed endpoints must require explicit user action,
  a valid Engine JWT, and server-side validation of product IDs, seller IDs,
  quantity limits, and payment/delivery choices.

### Minimal TypeScript Pattern

This example is provider-neutral. Replace `ai.proposeOrder` with OpenAI,
Anthropic, Groq, Ollama, Hugging Face, RunPod, or your own model service. The
important part is that the AI service returns an order proposal first; it does
not create the order until the app sends explicit user confirmation.

```ts
import {
  createYaatalClient,
  type CreateOrderRequest,
  type Product,
} from "@yaatal/client";

type AiOrderDraft = {
  productIds: string[];
  quantitiesByProductId: Record<string, number>;
  paymentMethod: "cash" | "wave";
  deliveryMethod: "pickup" | "bobo_managed";
  userFacingReason: string;
};

type AiProvider = {
  proposeOrder(input: {
    userMessage: string;
    products: Array<
      Pick<Product, "id" | "merchant_id" | "name" | "price_cents" | "stock">
    >;
  }): Promise<AiOrderDraft>;
};

export async function proposeOrderWithAi(input: {
  engineApiUrl: string;
  userJwt: string;
  userMessage: string;
  ai: AiProvider;
}) {
  const engine = createYaatalClient({
    baseUrl: input.engineApiUrl,
    token: input.userJwt,
  });

  const results = await engine.search.products({
    q: input.userMessage,
    limit: 10,
  });

  const products = results.products.filter((product) => product.stock > 0);
  const draft = await input.ai.proposeOrder({
    userMessage: input.userMessage,
    products: products.map((product) => ({
      id: product.id,
      merchant_id: product.merchant_id,
      name: product.name,
      price_cents: product.price_cents,
      stock: product.stock,
    })),
  });

  const allowedProducts = new Map(products.map((product) => [product.id, product]));
  const selected = draft.productIds
    .map((productId) => allowedProducts.get(productId))
    .filter((product): product is Product => product !== undefined);

  const firstProduct = selected[0];
  if (firstProduct === undefined) {
    return {
      kind: "no_safe_proposal" as const,
      message: "No in-stock Engine products matched the AI draft.",
    };
  }

  const sellerId = firstProduct.merchant_id;
  const sameSellerItems = selected.filter((product) => product.merchant_id === sellerId);

  const proposal: CreateOrderRequest = {
    seller_id: sellerId,
    payment_method: draft.paymentMethod,
    delivery_method: draft.deliveryMethod,
    items: sameSellerItems.map((product) => ({
      product_id: product.id,
      quantity: Math.max(
        1,
        Math.min(draft.quantitiesByProductId[product.id] ?? 1, product.stock, 10),
      ),
    })),
  };

  return {
    kind: "order_proposal" as const,
    requiresUserConfirmation: true,
    reason: draft.userFacingReason,
    products: sameSellerItems,
    proposal,
  };
}

export async function createOrderAfterUserConfirmation(input: {
  engineApiUrl: string;
  userJwt: string;
  proposal: CreateOrderRequest;
  confirmedByUser: boolean;
}) {
  if (!input.confirmedByUser) {
    throw new Error("Order creation requires explicit user confirmation.");
  }

  const engine = createYaatalClient({
    baseUrl: input.engineApiUrl,
    token: input.userJwt,
  });

  return engine.orders.create(input.proposal);
}
```

Recommended HTTP shape:

```text
POST /ai/order-proposals
  returns proposal only, never writes an order

POST /orders/confirmed
  requires user click/tap confirmation and valid Engine JWT
  validates proposal again
  calls client.orders.create(...)
```

### Provider Notes

- OpenAI, Anthropic, Groq, Hugging Face, and RunPod: call the provider from the
  AI service, never from Engine.
- Ollama or another local model: the AI service can call a local URL, then use
  `@yaatal/client` exactly the same way.
- Custom service: implement the same provider-neutral interface and return
  structured drafts. Keep Engine as the only system of record.
- Retrieval and ranking can be AI-assisted, but product IDs, seller IDs, prices,
  stock, order ownership, and delivery permissions must come from Engine.

## Francais

### Contrat V1

`@yaatal/client` V1 est le client de controle pour Engine. Il couvre auth,
produits, commandes, livraison, commerce BOBO, recherche, notifications et
analytics. Il n'expose volontairement pas `client.ai` en V1.

L'IA est externe et libre. Le service IA possede l'inference, les prompts, les
prompts de retrieval, le choix d'outils et la mise en forme des reponses. Engine
possede l'identite, l'autorisation, l'etat commerce, l'etat livraison, les
notifications, les evenements analytics et toutes les actions qui changent
l'etat metier.

### Architecture

Utilisez un de ces deux chemins, souvent les deux dans la meme app:

```text
App -> service IA -> @yaatal/client -> Engine
App -> @yaatal/client -> Engine
```

Le chemin assiste par IA sert a aider la recherche, resumer, recommander,
preparer un panier brouillon et proposer une commande. Le chemin direct app sert
aux ecrans normaux, au fallback et aux actions confirmees.

Gardez les details du fournisseur modele derriere votre service IA:

```text
App
  envoie le message utilisateur et le JWT Engine
service IA
  appelle OpenAI, Anthropic, Groq, Ollama, Hugging Face, RunPod ou un modele prive
  appelle @yaatal/client avec le JWT Engine
Engine
  valide le JWT, applique les droits, ecrit l'etat metier seulement apres confirmation
```

N'envoyez pas le JWT Engine au fournisseur modele. Le service IA peut utiliser
le JWT pour appeler Engine via `@yaatal/client`, mais les prompts envoyes aux
fournisseurs doivent contenir seulement le contexte produit/commande minimum.

### Regles De Frontiere

- L'IA possede l'inference et les prompts.
- Engine possede les actions qui changent l'etat.
- L'identite JWT vient de l'auth Engine. Ne faites pas confiance a un
  `buyer_id`, `seller_id` ou role utilisateur genere par le modele.
- La confirmation utilisateur bloque l'argent, la creation de commande, la
  confirmation livraison, l'annulation commande, les mutations sensibles du
  profil, KYC, les changements d'adresse et toute action commerce/livraison.
- L'IA peut rechercher des produits, resumer les resultats, expliquer les
  compromis et preparer une proposition de commande.
- L'IA ne doit pas creer directement une commande non verifiee, confirmer une
  livraison, annuler une commande, soumettre KYC ou modifier des donnees profil
  sensibles.
- Analytics peut enregistrer des evenements assistes par IA, mais analytics
  n'est pas l'etat metier et ne prouve pas qu'une action a eu lieu.

### Checklist Ops

- `ENGINE_API_URL`: configurez le service IA et l'app sur la meme URL Engine
  locale, staging ou production.
- Gestion du token auth: passez le JWT Engine de l'utilisateur depuis l'app vers
  le service IA en `Authorization: Bearer <jwt>` quand le service IA doit lire
  des donnees Engine scopees. Ne loggez jamais les JWT, ne les stockez pas dans
  les prompts et ne les envoyez pas aux fournisseurs IA.
- Cles fournisseurs IA: gardez les cles seulement dans l'environnement du
  service IA, par exemple `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`,
  `GROQ_API_KEY`, `HUGGINGFACE_TOKEN`, `RUNPOD_API_KEY` ou une URL de modele
  local prive.
- CORS et origins autorisees: autorisez l'origine app a appeler le service IA.
  Autorisez les origins app et service IA a appeler Engine seulement si ce sont
  des clients navigateur attendus. Les appels serveur a serveur n'ont pas besoin
  de CORS navigateur.
- Rate limits: limitez app -> service IA, service IA -> fournisseur et service
  IA -> Engine. Separez les retries fournisseur des mutations Engine.
- Logging: redactez JWT, cles fournisseur, numeros de telephone, adresses,
  champs profil, references paiement et prompts bruts avec donnees personnelles.
- Fallback: si l'IA est indisponible, gardez le chemin direct app avec
  `client.search`, `client.products`, `client.orders` et `client.delivery`.
- Pas de mutations IA non verifiees: separez les endpoints de proposition et de
  confirmation. Les endpoints confirmes doivent exiger une action utilisateur
  explicite, un JWT Engine valide et une validation serveur des product IDs,
  seller IDs, limites de quantite et choix paiement/livraison.

### Pattern TypeScript Minimal

L'exemple plus haut est volontairement neutre. Remplacez `ai.proposeOrder` par
OpenAI, Anthropic, Groq, Ollama, Hugging Face, RunPod ou votre propre service
modele. Le point important: le service IA retourne d'abord une proposition de
commande. Il ne cree pas la commande tant que l'app n'a pas envoye une
confirmation utilisateur explicite.

Forme HTTP recommandee:

```text
POST /ai/order-proposals
  retourne seulement une proposition, n'ecrit jamais une commande

POST /orders/confirmed
  exige un clic/tap de confirmation utilisateur et un JWT Engine valide
  valide encore la proposition
  appelle client.orders.create(...)
```

### Notes Fournisseurs

- OpenAI, Anthropic, Groq, Hugging Face et RunPod: appelez le fournisseur depuis
  le service IA, jamais depuis Engine.
- Ollama ou autre modele local: le service IA peut appeler une URL locale, puis
  utiliser `@yaatal/client` de la meme facon.
- Service sur mesure: implementez la meme interface neutre et retournez des
  brouillons structures. Engine reste la seule source de verite.
- Le retrieval et le ranking peuvent etre assistes par IA, mais product IDs,
  seller IDs, prix, stock, ownership commande et permissions livraison doivent
  venir d'Engine.
